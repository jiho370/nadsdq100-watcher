## 1. 왜 v1 점수는 실패했나 — 데이터가 보여준 원인 5가지

### D1. 호라이즌 불일치 (주범)
팩터는 **6개월** forward-return IC로 선별·검증됐는데, 점수는 **1개월** 수익률에
캘리브레이션했다. 같은 데이터에서 포트폴리오 초과수익이 호라이즌별로:

| 보유 | 초과수익(net) | 비고 |
|---|---|---|
| 1m | +0.48%p | 신호가 거의 없음 |
| 3m | +2.57%p | |
| 6m | +4.88%p | 팩터 선별 기준 호라이즌 |
| 12m | +9.89%p | |

즉 이 팩터 군집은 "느리게 작동하는" 신호다. 1개월 단위로 자르면 신호 대비 노이즈가
압도하고, 실제로 분위표에서 **1~2분위(최저 점수)가 평균수익 1위**(1.46%/1.57%)였다 —
1개월 구간에서는 단기 반전(short-term reversal)이 지배한다는 교과서적 패턴.
Spearman ρ=-0.0008(p=0.87)은 이 불일치의 필연적 결과다.

### D2. 임의 합성가중이 부활함
v1 복합점수 = 0.5·팩터 + 0.3·추세 + 0.2·저변동성. 이 0.5/0.3/0.2는 근거 없는 고정값 —
제안서 5장이 금지한 "임의 가중합"을 캘리브레이션 앞단에서 재생산했다.
더 나쁜 것: 추세 성분에 든 mom12_1의 이 표본 IC는 **음수(-0.025)** 다. 예측력이
음인 성분에 30%를 배정해 팩터 신호를 상쇄했다.

### D3. 중첩 표본이 3단계 게이트를 관대하게 만듦 (이상 징후 — 반드시 보고)
리밸 63일 × 보유 126일 → 인접 이벤트의 수익률 구간이 50% 겹친다. 32개 이벤트의
독립 정보량은 실질 ~16개.

- DSR 재계산: T=32 → **0.982 (통과)**, T_eff=16 → **0.927 (미통과)**.
  즉 "3단계 통과" 판정 자체가 중첩 미보정의 산물이었다.
- PBO(38.6%)도 블록 경계에서 purging/embargo가 없어 과소추정이다.
- 워크포워드 OOS -0.5%p와 정합적: 전략의 통계적 유의성은 **경계선 이하**로 봐야 한다.

### D4. "최적 가중치"의 식별 불안정
IC 1위 gp_assets(0.083)가 그리드 최적조합에서 가중치 0. 선택 기준을 바꾸면 최적조합이
바뀐다(비교표 기준 shareholder_yield1·accruals2, PBO 최고샤프 기준 shareholder_yield2·roa1).
신호는 특정 정수 조합이 아니라 **"퀄리티+주주환원 군집" 수준**에서만 존재한다.
개별 조합을 고르는 행위 자체가 다중검정(665개 시행)을 키운다.

### D5. 레짐 의존
워크포워드 검증구간(최근 ~2.5년)에서 6M 초과 -0.5%p. 전체 기간 +4.9%p는 앞쪽 구간이
견인했다. 기존 문서의 "+20%p·승률 89.5%"는 재현되지 않았다(제안서 ① 의심 확정).
참고: mom12_1 IC 음수는 STRATEGY.md의 모멘텀 중심 설계와 정면 배치되는 관찰이다.

## 2. 재설계 — 원인별 대응

핵심 결정: **점수는 6m 호라이즌 기대값의 순위**로 정의한다. 사용자는 10년+ 장기투자자로
월 2~3회는 "진입 결정 빈도"이지 보유기간이 아니다. 1m 캘리브레이션은 폐기.

### 2.1 점수 구성 (D2·D4 대응)
1. **팩터 군집 점수 하나만 쓴다**: 훈련구간에서 IC>0인 퀄리티·주주환원·현금흐름 계열
   (후보: gp_assets, gross_margin, shareholder_yield, accruals, cop, roa, fcf_ev, op_margin)
   을 **IR-가중**(w_f = mean(IC_f)/std(IC_f), 음수면 0)으로 결합한 z-합성.
   - 정수 그리드 탐색 폐지 → 시행 수 665 → 1. 다중검정 예산이 극적으로 준다.
   - 가중치는 워크포워드: 각 시점 t의 가중치는 t 이전 스냅샷 IC만으로 산출(look-ahead 금지).
2. **추세·변동성은 점수에 넣지 않는다**: mom12_1(IC<0)은 제외. 저변동성·200MA는 기존
   시스템의 필터/게이트 역할 유지(점수와 직교 정보로 리포트에 별도 표기).
   52주 고점 근접도는 훈련구간 IC>0일 때만 군집에 포함(자동 판정, 하드코딩 금지).
3. AI 가감점 ±1은 v1 설계 유지(라이브 전용, 캘리브레이션 제외).

### 2.2 캘리브레이션 (D1 대응)
- 풀: PIT 유니버스 전 종목, 월간 스냅샷(21d), **forward 6m** 수익률.
- 스냅샷 내 백분위 → 10분위 → 분위별 {평균수익, 승률, 손익비, 최악, 표본수}.
- **전체 기간과 최근 5년을 분리 표기**(D5 대응) — 두 구간의 분위 순서가 다르면 그 자체를
  리포트에 경고로 노출.

### 2.3 검증 (D3 대응 — 게이트 강화, 협상 불가 항목)
| 검정 | v1 (문제) | v2 (수정) |
|---|---|---|
| Spearman | 47,434건 풀링 1회 (중첩·군집으로 p값 허위 정밀) | **스냅샷 단위**: 각 리밸일 ρ_t 계산 → 평균 ρ와 t-stat(독립 단위 = 스냅샷). 기준: ρ̄>0 & t≥2 (HLZ 준용 시 3) |
| PBO | CSCV, purging 없음 | 블록 경계 **embargo = ⌈보유일/리밸일⌉ 이벤트** 제외(purged CSCV) |
| DSR | T=이벤트수(중첩 무시) | **T_eff = T × rebal_days/hold_days** 병기, 판정은 T_eff 기준 |
| 추가 | 없음 | D10−D1 6m 평균수익차 > 왕복비용(0.1%p) + WF OOS 구간에서도 ρ̄>0 |

전부 통과해야 `display_allowed=true`. 하나라도 실패하면 점수 비노출(v1 게이트 구조 재사용).

### 2.4 표시 (통과 시)
- 점수 = 당일 후보의 군집 점수 백분위 × 10 (0~10 정수).
- 이메일 표기 예: "8점 — 과거 동일 분위(6개월 보유): 평균 +X.X%, 승률 XX%, 손익비 X.X
  (최근 5년: 평균 +Y.Y%)". 숫자는 전부 score_calibration.json 실측치에서 읽는다.
- 매도(처분 시급도) 점수는 매수 점수 검증 통과 후 대칭 설계로 별도 착수(후순위).

### 2.5 정직한 실패 경로
6m에서도 스냅샷 Spearman이 실패하면 → 3m·12m 시도(시도 횟수를 pbo_report에 기록해
다중검정 예산 관리). 전부 실패하면 **점수 체계 도입 보류**를 공식 결론으로 한다.
D3·D5를 감안하면 이 가능성은 실재한다. 그 경우 남는 가치: 분위표 자체를 "이 전략의
기대값은 이 정도"라는 캘리브레이션 리포트로 제공(점수 없이).

## 3. Sonnet 구현 지침 (파일 단위)

1. `score_calibration.py` 수정 (발송 로직 무관, 독립 스크립트):
   - `--horizon 6m` 기본값 변경. COMP_W 고정가중 삭제.
   - `build_pool`: 워크포워드 IR-가중 군집 점수(2.1)로 교체. 스냅샷별 (ρ_t) 저장.
   - `calibrate`: 스냅샷 단위 Spearman(2.3), 전체/최근5년 분리 테이블(2.2).
   - 출력 스키마에 `spearman_by_snapshot{rho_mean,t_stat,n_snapshots}`,
     `recent_5y_deciles`, `attempted_horizons` 추가.
2. `overfit_stats.py` 수정:
   - `pbo_cscv(..., embargo:int)` — IS/OOS 경계 인접 이벤트 embargo개 제외.
   - `deflated_sharpe(..., t_eff)` — T_eff 병기, `passed`는 T_eff 기준으로 변경.
3. `backtest_costs.py`: trial_returns.json에 `rebal_days`,`hold_days` 메타 추가(2번이 T_eff
   계산에 사용). 그 외 불변.
4. 실행 순서(PC):
   ```
   py backtest_costs.py --years 10 --oos 0.4
   py overfit_stats.py                        # embargo·T_eff 반영 게이트
   py score_calibration.py --years 10        # 기본 6m, 게이트 통과 시에만
   ```
5. 검증 self-test: 합성 단조신호→통과 / 노이즈→차단 / 중첩 데이터에서 T_eff<T 확인.

## 부록 A — 지지·저항(S/R)의 검증 가능한 정의와 테스트 설계

원칙: "많이들 쓴다"는 기각 사유도 채택 사유도 아니다. 다수가 같은 선을 보면
자기실현적으로 작동할 수 있다는 것이 가설이고, 판정은 백테스트가 한다.
단, 테스트하려면 사람 눈이 아니라 **코드로 재현 가능한 정의**가 필요하다.

### A1. 코드화 가능한 S/R 신호 정의 (전부 종가·거래량만으로 계산)
| 신호 | 정의 | 비고 |
|---|---|---|
| `sr_support_dist` | 최근 252d 국소 저점들(±2% 밴드로 군집화, 터치 2회 이상) 중 현재가 아래 최근접 레벨까지 거리 % | 터치 횟수를 강도 가중 |
| `sr_resist_dist` | 위와 대칭(국소 고점 군집) — 현재가 위 최근접 저항까지 거리 % | 저항 돌파 직후 여부 플래그 포함 |
| `hi52_prox` | 52주 고점 근접도 | 기존 신호(앵커링의 대표형) |
| `lo52_prox` | 52주 저점 근접도 | 지지 앵커의 대표형 |
| `ath_prox` | 역사적 신고가 근접(52주 고점=역사고점 여부) | 단일 앵커일 때 효과 2~3배 보고됨 |
| `round_prox` | 라운드넘버(1·2·5×10^k)까지 거리 % | 심리적 레벨 |
| `vol_poc_dist` | 최근 252d 거래량 최대 가격대(POC)까지 거리 % | 거래량 프로파일 지지 |

### A2. 두 갈래로 나눠 테스트 (혼동 금지)
**(a) 횡단면 팩터로서** — "이 신호가 종목 간 수익률 순위를 예측하는가":
A1 신호를 `CLUSTER_CANDIDATES`와 별도 리스트(`SR_CANDIDATES`)로
`score_calibration.py` v2 파이프라인에 투입, 동일 게이트(스냅샷 IC·Spearman) 적용.
시행 수 7개 추가는 `attempted` 로그에 기록(다중검정 예산).

**(b) 실행 규칙으로서** — "진입·청산 타이밍을 개선하는가" (분할매수/분할매도의 본령):
`backtest_exec.py`(신규, 발송 로직 무관)로 **동일 종목·동일 신호** 조건에서 규칙만 비교:
- 진입: ①신호일 익일 전량 ②지지밴드(sr_support 또는 MA20 −2%) 눌림 대기 2분할(현행 entry_plan 방식) — 미체결 리스크 포함 평가
- 청산: ①현행 고정 트레일링 -20% 전량 ②ATR 2단 분할매도(피크 대비 -1.5×ATR60 절반, -2.5×ATR60 또는 200MA -3% 전량) ③지지 이탈(최근접 지지 -2%) 분할
- 평가지표: net 수익(비용 CostModel 재사용), 회전율, 손절 빈도, MDD, 미체결률.
  판정도 동일 프레임: 규칙 조합 수를 기록하고 PBO/DSR 적용.

### A3. 예상 관리
S/R이 (a)에서 유의할 가능성은 낮게 보나(앵커 계열 외), (b)에서는 비용·손절빈도를
낮추는 방향으로 기여할 수 있다. 어느 쪽이든 수치가 결정하며, 실패 기록도 남긴다.

### B3. 한국
KRX 업종지수(pykrx)로 B2와 동일 신호 산출 가능. '광통신' 같은 테마 분류는 무료
데이터에 없음 → 업종 근사 + (원하면) 수동 테마 매핑 파일(`theme_map_kr.json`)을
지호 님이 관리하는 옵션.
### B4. 정직한 경고
붐은 끝이 험하다(진입보다 청산이 성패를 좌우). B2가 매수를 돕는 만큼, 부록 A2-(b)의
청산 규칙과 반드시 결합 검증한다. "확 오를 때 올라타기"를 종목 단위 테마추격으로
구현하는 것은 이 시스템의 원칙(월 2~3회, 저회전, 검증 우선)과 충돌하므로 하지 않는다.
